
  CISB TEST — VARIABLE LENGTH ARRAYS (VLA) & alloca() STACK EXHAUSTION
  Full test package: code, commands, assembly analysis, statistics

This test demonstrates how unbounded stack allocation (via VLAs or alloca())
can exhaust the stack and skip the guard page, leading to a "stack clash" that
corrupts memory. It follows the same methodology as the null pointer and strict
aliasing tests.

NOTE

This test differs from check-removal bugs in one key way:

  - This is NOT the compiler removing a security check.
  - It is the LANGUAGE/RUNTIME having no failure signal for stack allocation.
  - malloc() returns NULL on failure (checkable). A VLA or alloca() that is
    too large simply grows the stack past the guard page with NO error,
    silently corrupting whatever memory lies beyond.

So the "fixes" fall into three categories:
  - DETECTION  : -Wvla  (compile-time warning that a VLA exists)
  - MITIGATION : -fstack-clash-protection  (runtime: probe each page so a
                 huge growth reliably hits the guard page = clean crash)
  - REAL FIX   : code change to bounded allocation (fixed buffer or checked malloc)


  COMPILING ALL VERSIONS

    # Record compiler version for the report
    gcc --version

    #  Vulnerable VLA, no protection 
    gcc -O2 -o vla_vuln              vuln_vla.c

    #  Vulnerable VLA, with -Wvla (DETECTION only, it prints a warning) 
    gcc -O2 -Wvla -o vla_warned      vuln_vla.c
    # Note: this STILL produces a vulnerable binary; -Wvla only warns.

    # Vulnerable VLA, with stack-clash protection (MITIGATION) 
    gcc -O2 -fstack-clash-protection -o vla_clash  vuln_vla.c

    # Vulnerable alloca, no protection 
    gcc -O2 -o alloca_vuln           vuln_alloca.c

    # Vulnerable alloca, with stack-clash protection 
    gcc -O2 -fstack-clash-protection -o alloca_clash  vuln_alloca.c

    # Safe fixes (no special flags needed) 
    gcc -O2 -o fixed_buf             fixed_buffer.c
    gcc -O2 -o fixed_malloc          fixed_malloc.c


 RUNNING THE TESTS


The argument is the allocation size in bytes. Small = safe, huge = exhaustion.

    # Normal small allocation: everything works 
    ./vla_vuln 64
    ./alloca_vuln 64
    ./fixed_buf 64
    ./fixed_malloc 64

    # Huge allocation: trigger the stack exhaustion 
    # 8 GB request - far beyond the ~8 MB default stack
    ./vla_vuln 8000000000          # CRASH: segfault, no clean error
    ./alloca_vuln 8000000000       # CRASH: segfault, no clean error

    # Same huge allocation with stack-clash protection 
    ./vla_clash 8000000000         # CLEAN crash at guard page (controlled)
    ./alloca_clash 8000000000      # CLEAN crash at guard page (controlled)

    # Same huge allocation, safe fixes 
    ./fixed_buf 8000000000         # REJECTED gracefully: "exceeds limit"
    ./fixed_malloc 8000000000      # REJECTED gracefully: malloc returns NULL

 DETECTION: THE -Wvla WARNING

    # Compile the VLA version with -Wvla and capture the warning
    gcc -O2 -Wvla -o vla_warned vuln_vla.c 2>&1 | tee wvla_warning.txt

  Expected warning (more or less this):
    vuln_vla.c: In function 'process':
    vuln_vla.c:XX:9: warning: variable length array 'buffer' [-Wvla]

  -Wvla is a COMPILE-TIME DETECTION mechanism. It does
  NOT change the binary. The vla_warned binary is exactly as vulnerable as
  vla_vuln. To confirm:

    # The two binaries should be functionally identical
    cmp vla_vuln vla_warned && echo "binaries identical (warning only)"

  The fixed versions produce NO -Wvla warning because they contain no VLA:
    gcc -O2 -Wvla -o /dev/null fixed_buffer.c   # silent
    gcc -O2 -Wvla -o /dev/null fixed_malloc.c   # silent


 ASSEMBLY ANALYSIS

Generate assembly for the key versions:

    gcc -O2 -S -o asm_vla.s        vuln_vla.c
    gcc -O2 -fstack-clash-protection -S -o asm_vla_clash.s  vuln_vla.c
    gcc -O2 -S -o asm_fixed_buf.s  fixed_buffer.c

WHAT TO LOOK FOR

THE VLA STACK ADJUSTMENT (vuln_vla.c at -O2)
    The stack pointer is moved by a RUNTIME value (a register), not a constant:

        ...
        mov    rax, rdi              ; n (the runtime size)
        ...
        sub    rsp, rax              ; <-- STACK GROWS BY VARIABLE AMOUNT
        ...                          ;     no probing of intermediate pages

    The danger: a single large 'sub rsp, rax' can move rsp past the guard
    page in one jump, landing in valid mapped memory beyond it.

WITH -fstack-clash-protection (asm_vla_clash.s)
    The compiler emits a PROBING LOOP that touches each page as the stack
    grows, so the guard page is always hit if the allocation is too large:

        ...
    .Lprobe_loop:
        sub    rsp, 0x1000           ; subtract ONE PAGE (4096 bytes)
        or     QWORD PTR [rsp], 0x0  ; <-- PROBE: touch the page
        cmp    rsp, <target>
        jne    .Lprobe_loop          ; repeat until full size allocated
        ...

    KEY OBSERVATION: the single 'sub rsp, rax' is replaced by a loop that
    probes every page. This guarantees the guard page is hit (clean crash)
    instead of being skipped (silent corruption).

FIXED BUFFER (asm_fixed_buf.s)
    The stack adjustment is a COMPILE-TIME CONSTANT:

        sub    rsp, 0x1010           ; <-- FIXED constant (MAX_BUFFER + frame)

    There is no runtime-variable stack growth at all. The allocation size
    is known at compile time and bounded.
